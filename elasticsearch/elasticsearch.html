title: Elasticsearch

Elasticsearch

HOME

  • Elasticsearch는 분산 검색 및 분석 엔진이다.

  • 모든 유형의 데이터에 대해 거의 실시간 검색 및 분석을 제공한다.

  • 데이터를 효율적으로 저장하고 빠른 검색을 지원하는 방식으로 인덱싱한다.

  • 데이터와 쿼리 볼륨이 증가하더라도 쉽게 확장이 가능하다.

  • Elasticsearch는 분산 문서 저장소이다.

  • JSON 문서로 데이터를 저장한다.

  • 클러스터에 다수의 노드가 있을 경우 문서는 클러스터 전반에 분산되어 저장된다.

  • 저장된 문서는 클러스터 내의 모든 노드에서 즉시 접근이 가능하다.

  • 저장된 문서에 대해 거의 실시간(1초 이내)으로 완전히 검색이 가능하다.

  • Elasticsearch는 역 인덱스 데이터 구조를 사용한다.

  • 역 인덱스 데이터 구조는 매우 빠른 '문서 전체 검색(full-text serachs)'을 지원한다.

  • 역 인덱스는 모든 도큐먼트에 존재하는 단어를 중복되지 않게 나열하고 각 단어가 포함된 모든 도큐먼트를 식별한다.

  • 인덱스는 최적화된 도큐먼트의 모음이다.

  • 도큐먼트는 데이터가 포함된 키-값 쌍인 필드의 모음이다.

  • Elasticsearch는 모든 필드의 모든 데이터를 인덱싱한다.

  • 인덱싱된 각 필드에는 최적화된 전용 데이터 구조가 있다.

  • 예를 들어 텍스트 필드는 역 인덱스에 저장되고 숫자 필드나 지리(geo) 필드는 BKD 트리에 저장된다.

  • 필드별 데이터 구조를 사용하기 때문에 빠른 검색이 가능하다.

  • Schema-less 기능을 가지고 있다.

  • 도큐먼트의 각 필드를 처리하는 방법을 명시적으로 지정하지 않고도 도큐먼트를 인덱싱 할 수 있다.

  • 동적 맵핑을 활성화 하면 자동으로 새로운 필드를 감지하고 인덱스에 추가한다.

  • 각 필드 값을 적절한 데이터 유형으로 감지하고 매핑한다.

  • 사용자가 동적 매핑을 제어하는 규칙을 정의하고 매핑을 명시적으로 정의하여 필드 저장 및 인덱싱 방법을 완전히 제어할 수 있다.

  • 사용자 매핑 정의를 통해 할 수 있는것

    • 전체 텍스트 문자열 필드와 정확한 값 문자열 필드 구분
    • 언어별 텍스트 분석 수행
    • 부분 일치를 위해 필드 최적화
    • 사용자 지정 날짜 형식 사용
    • 자동으로 감지할 수 없는 geo_point 및 geo_shape와 같은 데이터 유형 사용

  • 동일한 필드를 다른 목적으로 다른 방식으로 인덱싱하는 것이 유용한 경우가 많다.

  • 인덱싱 중에 문서 전체(full-text) 필드에 적용되는 분석체인은 검색시에도 사용된다.

  • 문서 전체(full-text) 필드를 쿼리할 때 쿼리 텍스트는 인덱스에서 용어가 검색되기 전에 동일한 분석을 거친다.

  • Apache Lucene 검색 엔진 라이브러리에 구축된 전체 검색 기능에 쉽게 엑세스가 가능하다.
  • 클러스터를 관리하고 데이터를 인덱싱하고 검색할 수 있는 REST API를 제공한다.
  • 사용자의 애플리케이션에서 다양한 언어로 Elasticsearch Client를 사용할 수 있다.

  • 구조화된 쿼리, 전문(full text) 쿼리, 복합 쿼리를 지원한다.

  • 구조화된 쿼리는 SQL에서 구성할 수 있는 쿼리 유형과 유사하다.

  • 전문(full-text) 쿼리는 쿼리 문자열과 일치하는 모든 문서를 찾아 검색어와 얼마나 잘 일치하는지 관련성별로 정렬하여 반환한다.

  • 구문 검색, 유사도 검색, 접두사 검색을 수행할 수 있다.

  • 자동 완성 제안을 받을 수 있다.

  • 고성능(을 요구하는?)의 지리 및 수치 쿼리를 지원하는 최적화된 데이터 구조에서 비 텍스트 데이터를 인덱싱한다.

  • 포괄적인 JSON-style 쿼리 언어(Query DSL)를 사용하여 모든 검색 기능을 사용할 수 있다.

  • SQL 스타일 쿼리를 구성하여 Elasticsearch 내부에서 기본적으로 데이터를 검색하고 집계할 수 있다.

  • JDBC 및 ODBC 드라이버를 사용하면 SQL을 통해 Elasticsearch와 상호작용 할 수 있다.

확장성 및 복원력: 클러스터, 노드 및 샤드

  • 항상 사용할 수 있도록 설계되어있다.

  • 필요에 따라 확장이 가능하다.

  • 본질적으로 분산되어있다.

  • 클러스터에 노드를 추가하여 용량을 늘릴 수 있다.

  • 사용 가능한 모든 노드에 데이터와 쿼리 로드를 자동으로 분산한다.

  • 확장성과 고가용성을 제공하기위해 각 노드들의 균형을 맞춘다.

  • 노드가 많을 수록 더 좋다.

  • 인덱스는 하나 이상의 물리적 샤드로 구성된 논리적 그룹일 뿐이다.

  • 샤드는 실제로 독립형 인덱스로 볼 수 있다.

  • 도큐먼트를 여러 샤드에 걸쳐 분산함으로써 인덱스에 배포한다.

  • 샤드를 여러 노드에 분산함으로써 중복성을 보장한다.

  • 이는 하드웨어 장애로부터 데이터를 보호한다. 또한 노드가 클러스터에 추가됨에 따라 쿼리 처리 능력을 증가시킨다.

  • 클러스터가 성장하거나 축소됨에 따라 elasticsearch는 자동으로 샤드를 마이그레이션하여 클러스터의 균형을 재조정한다.

  • 샤드에는 기본 샤드와 복제 샤드 두 가지 유형이 있다.

  • 인덱스의 각 도큐먼트는 하나의 기본 샤드에 분산된다.

  • 복제 샤드는 기본 샤드의 사본이다.

  • 복제 샤드는 하드웨어 오류로부터 데이터를 보호하기 위해 데이터의 중복 사본을 제공한다.

  • 복제 샤드는 검색이나 문서 조회 같은 읽기 요청 처리 능력을 향상시킨다.

  • 인덱스 내 기본 샤드의 수는 인덱스를 생성할 때 고정되지만, 복제 샤드 수는 언제든지 변경할 수 있다.

  • 샤드 크기와 인덱스에 설정된 기본 샤드의 수에 따라 여러가지 성능 고려 사항과 트레이드 오프가 있다.

  • 샤드가 많을수록 단순히 해당 인덱스를 유지하는데 더 많은 오버헤드가 발생한다.

  • 샤드 크기가 클수록 elasticsearch가 클러스터의 균형을 재 조정해야 할 때 샤드를 이동하는데 더 오랜 시간이 걸린다.

  • 작은 샤드를 많이 쿼리하면 샤드당 처리 속도가 빨라지지만, 쿼리가 많을 수록 오버헤드가 더 많기 때문에 더 적은 수의 큰 샤드를 쿼리하는 것이 더 빠를 수 있다.

  • 상황에 따라 다르므로 사용자가 잘 조절해서 사용하는 것이 필요하다.

  • 샤드의 평균 크기를 수 GB에서 수 십 GB 사이로 유지하는 것을 목표로 해야한다.

  • gazillion 샤드 문제를 피해야한다.

  • 노드가 보유할 수 있는 샤드의 수는 사용 가능한 힙 공간에 비례한다.

  • GB당 샤드의 수는 20개 미만이어야 한다는 것이 일반적인 규칙이다.

  • 사용자의 데이터의 쿼리로 테스트하는 것이 최적의 구성을 결정하는 방법이다.

재난 발생 시

  • CCR(Cross Cluster Replication)를 제공한다.

  • CCR은 hot backup으로서 인덱스를 기본 클러스터에서 보조 원격 클러스터로 자동 동기화하는 방법을 제공한다.

  • 기본 클러스터에 장애가 발생하면 보조 클러스터가 대신한다.

  • CCR은 액티브-패시브이다.

  • 기본 클러스터의 인덱스는 액티브 리더 인덱스로 모든 쓰기 요청을 처리한다.

  • 보조 클러스터에 복제된 인덱스는 읽기 전용 팔로워이다.

관리 및 feeding

  • elasticsearch내의 보안, 모니터링 및 관리 기능을 통해 Kibana를 클러스터 관리를 위한 제어 센터로 사용할 수 있다.

  • 데이터 롤 업 및 인덱스 수명 주기 관리와 같은 기능을 통해 시간이 지남에 따라 데이터를 지능적으로 관리할 수 있다.

Cluster Architecture(클러스터 구조)

Cluster(클러스터)
  • 클러스터는 Node(노드, 서버)들이 집단을 이뤄 하나로 동작하는 구조의 단위를 의미한다.
  • 클러스터 내에서 각 노드들은 데이터와 저장, 검색, 분석 작업을 공유한다.
Node(노드)
  • 클러스터를 구성하는 각 서버를 의미한다.
  • Master Node(마스터 노드): 클러스터의 상태를 관리한다.
  • Data Node(데이터 노드): 데이터 저장 및 쿼리를 처리한다.
  • Coordinating Node(코디네이팅 노드): 쿼리를 다른 노드로 라우팅한다.
  • Ingest Node(인제스트 노드):
  • Machine Learning Node(머신 러닝 노드):
Index(인덱스)
  • 데이터의 집합을 의미한다. RDBMS의 데이터베이스와 유사한 개념으로 볼 수 있다.
Shard(샤드)
  • 인덱스를 물리적으로 분할한 단위이다.
  • 데이터를 물리적으로 저장하고 분산하는 기본적인 단위이다.
  • 확장성과 성능 유지를 가능하게 하는 중요한 개념이다.
  • 프라이머리 샤드와 레플리카 샤드로 나뉜다
Segment(세그먼트)
  • 샤드 내부에서 도큐먼트를 저장하는 데이터 구조이다.
  • 데이터를 저장하고 검색하기 위해 사용하는 기본적인 단위이다.
  • 불변성을 가진다.
  • 효율적인 검색을 위해 구조적으로 최적화되어있다.
Document(도큐먼트)
  • Elasticsearch에서 데이터를 저장하는 가장 작은 단위이다.
  • JSON(JavaScript Object Notation) 형식으로 저장, 표현된다.
  • 관계형 데이터베이스의 행(row)와 유사한 개념이다.
  • Elasticsearch의 검색 쿼리는 도큐먼트를 기준으로 수행된다.

분산형 아키텍처

  • 다수의 샤드(shard)에 데이터(document)를 분산하여 저장한다.
  • 다수의 샤드는 다수의 노드(nord)에 분산된다.(수평적 확장성, 성능 저하 방지)
  • 프라이머리(primary) 샤드와 동일한 레플리카(replica) 샤드를 다른 노드에 배치한다. (읽기 성능 개선, 가용성, 내결함성)
  • 각 노드에서 동시에 검색을 수행한다.
  • 결과적 일관성(eventual consistency) 채택 - 모든 복제본이 동시에 일관성을 유지하지 않을 수 있음을 의미한다. 쿼리 설정, 인덱싱 방식 설정, 낙관적 동시성 제어(optimistic concurrency control, OCC)를 통해 일관성을 확보할 수 있다.

분산 아키텍처

고가용성 및 내결함성

실시간 검색

  • 인덱싱을 통해 저장 이전에 문서를 분석하고 역색인을 생성하여 빠른 검색을 지원한다.
  • 문서의 분석은 '분석 -> 토큰화 -> 정규화 -> 불용어 제거 -> 형태소 분석(같은 의미 단어 통합)' 과정을 거친다.
  • 분석을 통해 필터링된 각 단어와 해당 단어가 포함된 문서의 아이디를 매핑하여 역색인을 생성 또는 수정한다.
  • refresh interval에 따라 주기적으로 인덱스를 업데이트하여 새로운 데이터를 검색 가능 상태로 만든다. refresh interval은 기본적으로 1초로 설정되어있다.
  • 캐싱(caching)을 통해 반복적인 쿼리에 대한 응답 시간을 단축시킨다.
  • 데이터가 저장되는 단위인 세그먼트가 불변의 성질을 갖기때문에 빠른 읽기 작업이 가능하다.
  • 전체 텍스트 검색(full-text search)를 지원하여 관련 문서를 빠르게 검색한다.
  • 동시성을 지원하여 다수의 클라이언트에 대응하는 환경에 유리하다.

유연한 데이터 모델링

  • 비정형 데이터를 지원한다 - JSON 형식을 따르는 데이터일 경우 여러 데이터 타입을 동시에 처리한다.
  • schemaless 특성
    • 정의된 스키마를 필요로 하지 않는다
    • 하나의 인덱스에 여러 형태(구조)의 데이터를 저장할 수 있다.
    • 자동적으로 데이터의 타입을 감지한다.
    • 새로운 필드를 기존 문서에 추가할 수 있다.
  • 하나의 인덱스는 단일 유형을 가진다(단일 유형이 단일 구조를 의미하지는 않는다).
  • 필드 매핑을 통해 문서의 구조와 각 필드의 데이터 타입을 정의할 수 있다.
  • 인덱스 템플릿을 사용하여 새로운 인덱스가 생성될 때 기본 매핑 및 설정을 미리 정의할 수 있다. 일관된 데이터 모델을 유지할 수 있다. 인덱스 이름 패턴을 기반으로 자동 적용된다.
  • 데이터가 인덱스에 추가될 때, 필드의 유효성을 검사하여 잘못된 데이터를 사전에 방지할 수 있다.

    • 빠른 검색
      • Apache Lucene 기반
      • 인덱스 사용
      • 역색인(Inverted Index) 구조 사용으로 특정 단어나 필드에 대한 검색이 빠름
    • RESTful API
      • JSON 기반의 RESTful API 제공으로 타 시스템과 쉬운 통합
      • JSON 포맷을 사용한 API요청으로 데이터 인덱싱 및 검색
      • 다양한 프로그래밍 언어를 위한 클라이언트 라이브러리 지원
    • 강력한 검색 및 필터링 기능
      • Boolean Queries, Range Queries, 텍스트 분석 등의 Query DSL(Query Domain Specific Language)을 통해 세부적인 검색 조건 설정 가능
      • 필터링, 정렬 기능 사용 가능
      • 자동 완성 및 제안 기능 제공
    • 다양한 분석 기능
      • 집계 기능을 사용하여 데이터 그룹화와 다각도 분석 가능
      • 메트릭(metric), 버킷(bucket) 집계로 평균값, 합계, 최대/최소값, 분포등 계산 가능
      • 복잡한 데이터 분석 쿼리를 효율적으로 처리(로그, 통계 분석 등에 자주 사용)
    • 확장성과 유연성
      • 수평적으로 노드를 추가하는 것으로 쉽게 확장 가능
      • 스키마리스(Schemaless)특정 제공으로 데이터 구조 정의 없이 데이터 저장 및 필요에 따라 필드 매핑 수정
      • 다양한 형식의 데이터 저장으로 전체 텍스트 검색 및 구조적 데이터 검색에 적합
    • 실시간 데이터 처리
      • 인덱싱 즉시 거의 실시간으로 검색 가능, 최신 데이터 즉각 반영 가능
      • 로그 모니터링 또는 이벤트 트레킹 등의 실시간 데이터 스트림 다루는 경우에 유용
    • Kibana와 통합
      • Kibana와 통합으로 데이터의 시각적 분석 및 대시보드 구성 가능
